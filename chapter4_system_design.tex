\chapter{System Design}
\label{ch:systemdesign}

\justifying
This chapter presents the comprehensive system design of the Automated Diagram Digitization platform, detailing the architectural decisions, data flow, database schema, API design, and module specifications. The system design follows industry best practices for web applications, ensuring scalability, maintainability, security, and seamless integration between the frontend editor, backend services, and AI processing engine.

\section{System Architecture Overview}

\justifying
The Automated Diagram Digitization platform is built on a robust three-tier architecture ensuring scalability, maintainability, and security while providing seamless integration between the React frontend, FastAPI backend, and PostgreSQL database. This architectural model separates the system into three logical layers: the presentation layer responsible for user interaction, the application layer that implements the business logic and AI processing, and the data layer that manages persistent storage.

Such separation of concerns allows each tier to operate independently while communicating through well-defined interfaces. This improves maintainability because modifications in one layer do not directly affect the others. It also enables future scalability, allowing the frontend, backend, and database components to be deployed or upgraded independently based on system demand.

\begin{figure}[ht]
    \centering
    \includegraphics[width=0.9\textwidth]{image/flow.jpg}
    \caption{Three-Tier System Architecture of Automated Diagram Digitization Platform}
    \label{fig:ch4-architecture}
\end{figure}

\subsection{Presentation Layer (Client Tier)}

\justifying
The presentation layer serves as the user interface through which all users interact with the platform. It utilizes React 18 with Vite for fast development and optimal performance, React Flow for interactive diagram rendering, and Tailwind CSS for responsive, modern design. Key advantages include component-based architecture for reusability, fast refresh during development, and a consistent experience across all devices and screen sizes.

\justifying
The interface is organized into two main sections: the input page with three tabs for image upload, text prompt, and JSON import; and the flowchart editor with a draggable shape sidebar, canvas area with background grid, and toolbar with undo/redo, auto-layout, export, and theme toggle controls.

This structured interface design allows users to move smoothly between diagram input and editing stages without confusion. The tab-based input system ensures that multiple methods of diagram generation remain easily accessible, while the editor environment provides tools for refining and customizing the generated flowcharts. By combining these elements within a single interface, the platform offers a consistent workflow from diagram creation to final export.

\subsection{Application Layer (Business Tier)}

\justifying
The application layer implements all business logic, validation, workflow orchestration, and AI integration using FastAPI with Python. Key components include RESTful API endpoints for image analysis and text-to-flowchart generation, integration with Google Gemini AI using custom prompts, request validation using Pydantic models, centralized error handling with regex fallback for malformed responses, and asynchronous database operations using SQLAlchemy.

This layer acts as the central processing unit of the system, coordinating communication between the frontend interface, the AI processing service, and the database. By handling request validation, error handling, and response formatting in a centralized manner, the backend ensures consistent system behavior and simplifies debugging and maintenance tasks.

\subsection{Data Layer (Database Tier)}

\justifying
The data layer manages all persistent storage using PostgreSQL with JSONB support for flexible schema storage. Key features include UUID v4 primary keys for distributed generation without coordination, JSONB columns for storing complete React Flow state (nodes and edges), timestamp tracking with timezone support for created\_at and updated\_at fields, foreign key constraints where applicable, and connection pooling for optimal performance.

The database layer also supports efficient querying and reliable storage of flowchart data even as the number of diagrams grows over time. PostgreSQL’s support for JSONB enables the system to store complex node and edge structures without requiring rigid relational schemas for every property. This flexibility allows the platform to support future enhancements such as additional node types, styling options, or metadata fields without requiring major database redesign.

\section{Data Flow Diagrams}

\justifying
Data Flow Diagrams (DFDs) provide a graphical representation of how information moves through the Automated Diagram Digitization platform. They illustrate the sequence of data transformations that occur when users interact with the system, from initial input to final diagram output. By visualizing the movement of data between different system components, DFDs help clarify how the frontend interface, backend services, AI processing engine, and database interact during each workflow.

\subsection{Image Upload and Analysis Flow}

\justifying
The image analysis process enables users to upload flowchart images for automatic digitization. The workflow proceeds as follows: the user selects an image file via drag-drop or file picker, the frontend validates file type and size, the image is sent to the backend via multipart/form-data, the backend processes the image using Pillow (resizing to thumbnail), the image is sent to Gemini AI with the system prompt, the AI returns JSON with nodes and edges, the backend parses the response and ensures edge IDs exist, and the frontend renders the flowchart in the editor. The user can then edit, save, or export the diagram.

\begin{figure}[ht]
    \centering
    \includegraphics[width=0.95\textwidth]{image/df.jpg}
    \caption{Image Upload and Analysis Data Flow Diagram}
    \label{fig:chapter4-image-flow}

\subsection{Text-to-Flowchart Generation Flow}

\justifying
The text-to-flowchart feature enables users to describe algorithms in natural language for automatic flowchart generation. The workflow proceeds as follows: the user enters a text description in the prompt textarea, the frontend sends the prompt to the backend via POST request, the backend combines the user prompt with the text-to-flowchart system prompt, the combined prompt is sent to Gemini AI, the AI returns JSON with nodes and edges (may include explanatory text), the backend extracts JSON using regex pattern matching, the parsed JSON is returned to the frontend, and the editor renders the generated flowchart. The user can then modify the generated diagram as needed.

\subsection{JSON Import and Export Flow}

\justifying
The JSON import/export functionality enables data portability and integration with other tools. For import: the user uploads a JSON file or pastes JSON into textarea, the frontend validates JSON format, the validated JSON is passed directly to the editor for rendering. For export: the user clicks the export button and selects format (JSON/PNG/SVG), for JSON export the current nodes and edges are serialized to JSON and downloaded as a file, for PNG export the canvas is captured using html-to-image and downloaded, for SVG export the React Flow SVG renderer generates vector graphics for download.

\section{Database Schema Design (PostgreSQL)}

\justifying
The Automated Diagram Digitization platform utilizes PostgreSQL with JSONB support, providing reliable data persistence, ACID compliance, and flexible schema for storing complex flowchart data. The schema uses UUID primary keys, timestamp tracking, and JSONB columns for flow data storage.

\begin{figure}[ht]
    \centering
    \includegraphics[width=\textwidth]{image/arcch.jpg}
    \caption{Database Entity-Relationship Diagram}
    \label{fig:schema}
\end{figure}

\justifying
\noindent
The Entity-Relationship diagram above illustrates the database structure. At the center is the \texttt{flowcharts} table, which stores all flowchart data. Each flowchart has a unique UUID identifier, a title string, a JSONB column containing the complete React Flow state (nodes and edges), and timestamp columns for creation and last update tracking.

\justifying
\noindent
The JSONB column provides flexibility to store varying node and edge structures without requiring schema changes. Each node object contains id, type, position coordinates, data (including label), width, height, and optional styling properties. Each edge object contains id, source, target, optional label, type (smoothstep/straight), markerEnd for arrows, and optional styling properties.

\justifying
\noindent
The schema follows best practices for web applications, uses \texttt{created\_at} and \texttt{updated\_at} timestamps with timezone support for audit compliance, applies indexes on frequently queried columns, and uses UUID primary keys for security benefits over sequential integers. The JSONB column supports indexing on specific fields if needed for performance.

\justifying
\noindent
All timestamp columns use \texttt{TIMESTAMPTZ} rather than a timezone-naive type, storing values in UTC and allowing the application layer to convert to the user's local timezone at render time. This is particularly important for a platform that may be deployed across different geographical regions. Consistent UTC storage eliminates ambiguity in timestamp comparisons.

\justifying
\noindent
The schema is managed through a versioned migration system using Alembic (SQLAlchemy migrations), capturing every structural change as an incremental script across development, staging, and production environments. This ensures consistent database evolution across all deployments.

\section{Module-wise Detailed Design}

\justifying
This section provides design specifications for each major module of the Automated Diagram Digitization platform, describing their internal components and interactions with other modules. The modular architecture enables each functional component of the system to operate independently while contributing to the overall workflow. By separating responsibilities across modules, the system becomes easier to maintain, extend, and debug.

Each module focuses on a specific task such as handling user input, processing AI-based recognition, editing diagrams, exporting results, or managing persistent storage. This modular approach ensures that enhancements or modifications to one part of the system can be implemented without significantly affecting other components.

\subsection{Input Module}

\justifying
The input module handles all user input methods before AI processing. Components include:

\begin{itemize}
    \item \textbf{Image Upload Component}: Drag-drop zone with file picker fallback, file type validation (PNG, JPG, WebP), file size validation, preview thumbnail, and upload progress indicator.

    \item \textbf{Text Prompt Component}: Multi-line textarea with placeholder text, character count (optional), example prompt button, and submit with loading state.

    \item \textbf{JSON Import Component}: Textarea for JSON paste with syntax highlighting, file upload button for .json files, JSON validation with error messaging, and format preview option.

    \item \textbf{Tab Navigation}: Tabbed interface switching between input methods with active state styling and form reset on tab change.
\end{itemize}

\subsection{AI Processing Module}

\justifying
The AI processing module interfaces with Google Gemini AI and handles response parsing. Components include:

\begin{itemize}
    \item \textbf{Gemini API Client}: Configured with API key from environment variables, handles authentication, and sends requests with appropriate system prompts.

    \item \textbf{Image Processor}: Uses Pillow to open uploaded images, resizes to thumbnail (1024x1024 max) for optimal API performance, and maintains aspect ratio.

    \item \textbf{Prompt Manager}: Stores and manages system prompts for image analysis and text-to-flowchart generation, combines user input with system prompts.

    \item \textbf{Response Parser}: Parses JSON responses from Gemini, handles malformed responses using regex extraction (\textbackslash{}(\textbackslash.*\textbackslash{})), ensures edge IDs exist (generates e\{source\}-\{target\} if missing), and validates required fields.

    \item \textbf{Error Handler}: Catches and logs API errors, returns user-friendly error messages, and provides fallback responses when possible.
\end{itemize}

\subsection{Editor Module}

\justifying
The editor module is the core interactive component for diagram creation and modification. Components include:

\begin{itemize}
    \item \textbf{React Flow Canvas}: Renders nodes and edges with background grid, supports zooming and panning, and handles selection and deletion.

    \item \textbf{Sidebar Palette}: Displays draggable shape icons (start, process, decision, io) with SVG icons, implements HTML5 drag-drop with data transfer, and provides visual feedback during drag.

    \item \textbf{Custom Node Components}: BaseNode with common functionality, ProcessNode with rectangle shape, StartNode with rounded rectangle (pill shape), DecisionNode with diamond shape (clip-path polygon), and InputNode with parallelogram shape (skew transform).

    \item \textbf{Node Resizer}: Uses React Flow's NodeResizer component, visible only when node selected, maintains minimum dimensions (80x40 for most nodes, 120x120 for decision), and preserves text readability.

    \item \textbf{Edge Label Editor}: Double-click handler on edges, inline input popup at mouse position, Enter key saves, Escape cancels, and blur saves automatically.

    \item \textbf{Undo/Redo System}: Snapshot-based history using useState, takeSnapshot before operations, undo restores previous state, redo restores next state, and keyboard shortcuts (Ctrl+Z, Ctrl+Y, Ctrl+Shift+Z).

    \item \textbf{Auto-Layout Engine}: Column-based clustering algorithm, calculates true center X for each node, clusters within tolerance (150px), snaps nodes to average column positions, and updates edge types based on alignment.

    \item \textbf{Styling Controls}: Color picker for nodes and edges, text styling buttons (bold, italic), font size dropdown (10-20px), and style application to selected elements.
\end{itemize}

\subsection{Export Module}

\justifying
The export module handles all diagram export functionality. Components include:

\begin{itemize}
    \item \textbf{PNG Exporter}: Uses html-to-image library to capture canvas element, creates download link with data URL, and names file using flowchart title.

    \item \textbf{SVG Exporter}: Leverages React Flow's built-in SVG export capabilities, generates vector graphics with proper scaling, and maintains styling.

    \item \textbf{JSON Exporter}: Serializes current nodes and edges to JSON string with pretty printing (2-space indent), creates blob with application/json type, and triggers download.

    \item \textbf{Export Menu}: Dropdown menu with export options, icons for each format, and disabled state during export operations.
\end{itemize}

\section{Algorithm Design}

\justifying
The Automated Diagram Digitization platform incorporates several internal algorithms to support diagram generation, layout organization, and editing functionality. These algorithms are responsible for maintaining structural consistency within generated diagrams and ensuring that editing operations remain efficient and reliable.

Algorithmic design plays a critical role in transforming raw AI-generated data into visually organized flowcharts. By applying structured layout techniques and maintaining editing history through snapshot mechanisms, the system provides users with both automated organization and flexible manual control during diagram creation.

\subsection{Column-based Auto Layout Algorithm}

\justifying
The column-based auto layout algorithm is responsible for organizing nodes in a flowchart so that they appear visually aligned and structured. When diagrams are generated automatically by the AI system or manually edited by users, node positions may not always be perfectly aligned. The auto layout algorithm corrects this by grouping nodes into vertical columns based on their horizontal positions.

The algorithm begins by determining the maximum width required for each node type. Node types such as start nodes, process nodes, decision nodes, and input/output nodes may have different sizes. By determining the largest width for each type, the algorithm ensures visual consistency across the diagram.

After establishing the node widths, the algorithm calculates the horizontal center position of each node using its current position and width. These center positions are then used to sort the nodes along the horizontal axis. Sorting enables the algorithm to identify nodes that are positioned near one another.

Once sorted, the algorithm performs a clustering process in which nodes whose center positions fall within a defined tolerance range are grouped together into a single column. Each column maintains an average center value calculated from all nodes assigned to that group.

In the final stage, the algorithm repositions each node so that its center aligns with the calculated average center of its column. This ensures that nodes belonging to the same logical stage of a flowchart appear vertically aligned while maintaining their original vertical ordering. As a result, the flowchart becomes visually balanced and easier to interpret.

\subsection{Undo and Redo Snapshot Algorithm}

\justifying
The platform includes an undo and redo mechanism that allows users to revert or restore modifications made while editing diagrams. This feature is implemented using a snapshot-based history management algorithm.

Whenever a user performs an operation that modifies the diagram, such as adding a node, deleting an edge, moving an element, or changing text content, the system captures a snapshot representing the complete state of the diagram at that moment. Each snapshot contains the full set of nodes and edges currently present in the flowchart.

Snapshots are stored in a history structure consisting of two stacks referred to as the \textit{past} stack and the \textit{future} stack. The past stack records previous states of the diagram, while the future stack stores states that can be restored through redo operations.

When a user performs an undo operation, the system retrieves the most recent snapshot from the past stack and restores the diagram to that earlier state. At the same time, the current state of the diagram is stored in the future stack, allowing it to be restored later if necessary.

If the user performs a redo operation, the system retrieves the next snapshot from the future stack and applies it to the diagram. The previously restored state is then returned to the past stack. This two-stack mechanism allows users to move backward and forward through their editing history without losing information.

Keyboard shortcuts such as \texttt{Ctrl+Z} for undo and \texttt{Ctrl+Y} or \texttt{Ctrl+Shift+Z} for redo provide quick access to these operations, improving usability during diagram editing.

\justifying
Together, these algorithms ensure that diagrams remain organized and editable while providing users with flexible control over the editing process. The auto layout algorithm enhances visual clarity by maintaining alignment among nodes, while the snapshot-based history system provides reliable recovery from editing mistakes.

\section{Error Handling Design}

\justifying
The system implements comprehensive error handling at multiple levels to ensure robustness.

\subsection{AI Response Error Handling}

\begin{itemize}
    \item \textbf{Malformed JSON}: Uses regex extraction to find JSON block within text response: \texttt{re.search(r'(\textbackslash{}\{\textbackslash{}.*\textbackslash{}\}\textbackslash{})', text, re.DOTALL)}.

    \item \textbf{Unescaped Characters}: Uses \texttt{json.loads(json\_str, strict=False)} to handle unescaped newlines within strings.

    \item \textbf{Missing Fields}: Validates required fields (nodes, edges) and provides defaults if missing.

    \item \textbf{Missing Edge IDs}: Automatically generates IDs in format \texttt{e\{source\}-\{target\}} if not provided by AI.
\end{itemize}

\subsection{API Error Handling}

\begin{itemize}
    \item \textbf{400 Bad Request}: Returned for missing prompt field, invalid JSON format, or validation failures.

    \item \textbf{404 Not Found}: Returned when flowchart ID does not exist in database.

    \item \textbf{500 Internal Server Error}: Returned for AI API failures, database errors, or unexpected exceptions with error logging.
\end{itemize}

\subsection{Frontend Error Handling}

\begin{itemize}

\section{Performance Optimization}

\justifying
The Automated Diagram Digitization platform implements multiple optimization strategies to ensure responsive user experiences.

\textbf{Backend Optimization:}
\begin{itemize}
    \item \textbf{Image Resizing}: Thumbnail resizing to 1024x1024 max before sending to Gemini API reduces payload size and processing time.

    \item \textbf{Connection Pooling}: SQLAlchemy connection pooling with configurable pool size (5-20 connections) minimizes connection overhead.

    \item \textbf{Async Operations}: FastAPI async endpoints with await for database and AI operations improve concurrency.

    \item \textbf{JSONB Indexing}: GIN indexes on flow\_data for potential future querying requirements.
\end{itemize}

\textbf{Frontend Optimization:}
\begin{itemize}
    \item \textbf{React Flow Performance}: Uses virtualization for large diagrams, limits re-renders with memo and useCallback.

    \item \textbf{Debounced Operations}: Auto-save and auto-layout debounced to prevent excessive operations during rapid edits.

    \item \textbf{Lazy Loading}: Code splitting with React.lazy for editor component (loaded only when needed).

    \item \textbf{CSS Variables}: Theme switching uses CSS variable updates instead of full re-renders.
\end{itemize}

\textbf{AI Optimization:}
\begin{itemize}
    \item \textbf{Prompt Engineering}: Temperature=0 for deterministic outputs, response\_mime\_type="application/json" for structured responses.

    \item \textbf{Image Optimization}: Thumbnail resizing reduces API latency and costs.

    \item \textbf{Caching}: Consider caching frequent prompts or responses for future implementation.
\end{itemize}
