\chapter{Implementation}
\label{ch:implementation}

This chapter details the implementation of the Automated Diagram Digitization system, describing each module's logic, API interactions, and integration with the broader application. The implementation follows a layered architecture: FastAPI handles HTTP routing, PostgreSQL serves as the persistent store, and React with Vite presents the user interface. All code is organised within separate frontend and backend directories, with environment-specific settings separated into a \texttt{.env} file.

% -----------------------------------------------------------------------------
\section{Development Environment Setup}
\label{sec:dev-env}

The backend runs on Python 3.9+ with the FastAPI framework. The database is a PostgreSQL 16 instance with JSONB support for flexible schema storage. The frontend runs on Node.js v13+ with React 19 and Vite for fast development. For simplified orchestration, the system is containerized using Docker, allowing all three services (database, backend, frontend) to be provisioned through a single command. Sensitive configuration values -- database connection strings and Gemini API keys -- are stored in a \texttt{.env} file at the project root and loaded via the \texttt{python-dotenv} package.

The development workflow was organized using separate frontend and backend directories to maintain modularity. To ensure environment parity, the entire stack can be orchestrated using \textbf{Docker} and \textbf{Docker Compose}, providing isolated containers for the frontend, backend, and PostgreSQL database. Alternatively, the frontend can be managed via Node.js and npm, while the backend uses a Python virtual environment to isolate project libraries. This flexible setup ensures that dependencies remain controlled and reproducible across different development machines.

Version control was maintained using Git, with the project repository hosted on GitHub. This enabled structured version tracking, collaborative development, and integration with deployment platforms. Source code changes were tested locally before being committed to the repository, ensuring stability of the application during development.

\section{Backend Implementation}
\label{sec:backend}

The backend of the Automated Diagram Digitization platform is responsible for handling all core processing tasks, including AI-based diagram recognition, data validation, database interaction, and communication with the frontend interface. Implemented using the FastAPI framework, the backend exposes a set of RESTful endpoints that allow the frontend application to submit images or text prompts and receive structured flowchart data in JSON format.

The backend architecture emphasizes modularity and reliability. Each endpoint performs a clearly defined task, while shared utilities handle operations such as image preprocessing, prompt management, and response validation. This design simplifies maintenance and allows the backend to scale efficiently as additional features are introduced.

\subsection{Image Analysis Endpoint}

The image analysis endpoint accepts an uploaded flowchart image, processes it, sends it to Gemini AI with a custom system prompt, and returns structured JSON containing nodes and edges. The image is resized to a thumbnail of 1024x1024 pixels for optimal API performance while maintaining aspect ratio. By leveraging Gemini's multimodal capabilities, the system accurately extracts both textual labels and geometric structures in a single pass, ensuring that spatial relationships are preserved during reconstruction. To prevent malformed data from affecting the editor, the endpoint validates the AI's response against a predefined schema and utilizes a heuristic parsing layer for error recovery when necessary. The core logic of the image analysis endpoint is shown in \textit{Listing~\ref{lst:analyze_endpoint}: Image Analysis Endpoint -- Core Logic}.

\begin{lstlisting}[language=Python,
    label={lst:analyze_endpoint},

@app.post("/analyze/image")
async def analyze_flowchart(file: UploadFile = File(...)):

    # Step 1: Read and open image
    contents = await file.read()

    # Step 4: Parse and return

\subsection{Text-to-Flowchart Endpoint}

The text-to-flowchart endpoint accepts a natural language description of an algorithm and returns a generated flowchart. The user prompt is combined with a specialized system prompt designed for flowchart generation. The response is cleaned using regex extraction to handle any explanatory text the AI might add, ensuring only valid JSON is returned. By mapping semantic logic directly to node and edge attributes, the system ensures that the generated flowchart accurately represents the described algorithm's control flow.

\subsection{Database Operations}

The database layer uses SQLAlchemy with async support for PostgreSQL operations. The flowcharts table stores each diagram with a UUID primary key, title, JSONB column for the complete React Flow state, and timestamps for creation and updates. The JSONB column provides flexibility to store varying node and edge structures without requiring schema changes. The use of asynchronous database drivers minimizes latency during high-traffic sessions, ensuring that diagram saving and retrieval operations remain nearly instantaneous for the end-user.

The frontend of the Automated Diagram Digitization system provides the interactive environment through which users create, edit, and export diagrams. Built using React~18 with Vite, the frontend focuses on delivering a responsive and intuitive user experience while maintaining high performance for interactive diagram editing. The use of Vite ensures near-instantaneous build times and optimized asset loading, contributing to the overall fluidity of the application's interface.

This component-based structure simplifies interface maintenance and allows the system to be extended with additional features in the future.

\subsection{Undo/Redo Functionality}

Undo and redo operations are implemented using keyboard controls to allow quick correction of editing actions within the flowchart editor. The system listens for standard keyboard shortcuts and updates the diagram state accordingly. Pressing \texttt{Ctrl+Z} performs an undo operation, restoring the diagram to its previous state, while \texttt{Ctrl+Y} or \texttt{Ctrl+Shift+Z} performs a redo operation to reinstate the most recently undone action.

The implementation relies on maintaining a history of diagram states. Whenever a modification occurs -- such as node movement, edge creation, deletion, or label editing -- the previous state of the nodes and edges is stored. During an undo operation, the editor restores the most recent saved state, ensuring that accidental changes can be reversed quickly. This approach improves usability and enables users to experiment with diagram modifications without the risk of permanently losing previous configurations.

\subsection{Auto-Layout Feature}

The auto-layout feature utilizes a custom \textbf{Column Snapping (1D Clustering)} algorithm, implemented within the system's core flowchart hook, to intelligently organize nodes into a readable and aligned format. The implementation scans existing node positions and groups nodes that share a similar horizontal axis within a specific \texttt{TOLERANCE} of 150px. The algorithm then "snaps" all nodes in a group to their calculated average center and normalizes node widths based on their type (e.g., \texttt{process}, \texttt{decision}). This ensures that diagrams remain visually consistent even after complex manual edits. Furthermore, the layout engine dynamically adjusts edge routing, switching to \texttt{straight} connections for nodes aligned within a 50px threshold to create a clean, professional look. \textit{Refer Figure~\ref{fig:autolayout-before}: Flowchart before applying auto-layout and Figure~\ref{fig:autolayout-after}: Flowchart after applying auto-layout.}

The core logic for the column snapping layout engine is implemented through two primary functions: \texttt{normaliseAndSnap} and \texttt{applyAutoLayout}. The first function handles the clustering and alignment of nodes based on proximity, while the second propagates these changes and optimizes the visual style of connecting edges. By combining grouping with type-based width normalization, the system maintains high graphical fidelity without requiring complex hierarchical calculations. \textit{Refer Figure~\ref{lst:dagre-layout}: Implementation of the auto-layout engine.}

\begin{lstlisting}[language=JavaScript, caption={Implementation of the auto-layout engine.},basicstyle=\scriptsize\ttfamily, float=!htbp, label=lst:dagre-layout]
const normaliseAndSnap = (nodes) => {
  const maxWidths = { start: 100, process: 150, io: 150, decision: 120 };
  nodes.forEach(node => {
    const nodeWidth = node.width || (node.style && node.style.width) || maxWidths[node.type];
    if (nodeWidth > maxWidths[node.type]) maxWidths[node.type] = nodeWidth;
  });

  const TOLERANCE = 150;
  const columns = [];
  const nodesWithCenters = nodes.map(node => {
    const finalWidth = maxWidths[node.type];
    const trueCenterX = node.position.x + finalWidth / 2;
    return { ...node, trueCenterX, finalWidth };
  });

  nodesWithCenters.sort((a, b) => a.trueCenterX - b.trueCenterX).forEach(node => {
    const cx = node.trueCenterX;
    const existingCol = columns.find(col => Math.abs(col.averageCenterX - cx) < TOLERANCE);
    if (existingCol) {
      existingCol.centerValues.push(cx);
      existingCol.averageCenterX = existingCol.centerValues.reduce((s, v) => s + v, 0) / existingCol.centerValues.length;
    } else {
      columns.push({ centerValues: [cx], averageCenterX: cx });
    }
  });

  return nodesWithCenters.map(node => {
    const myCol = columns.find(col => Math.abs(col.averageCenterX - node.trueCenterX) < TOLERANCE);
    return {
      ...node,
      style: { ...node.style, width: node.finalWidth },
      position: { x: myCol.averageCenterX - node.finalWidth / 2, y: node.position.y }
    };
  });
};

const applyAutoLayout = useCallback(() => {
  setNodes((currentNodes) => {
    const newlyAlignedNodes = normaliseAndSnap(currentNodes);
    setEdges((currentEdges) => currentEdges.map(edge => {
      const srcNode = newlyAlignedNodes.find(n => n.id === edge.source);
      const tgtNode = newlyAlignedNodes.find(n => n.id === edge.target);
      const isVertical = Math.abs(srcNode.position.x - tgtNode.position.x) < 50;
      return { ...edge, type: isVertical ? "straight" : "smoothstep" };
    }));
    return newlyAlignedNodes;
  });
}, [setNodes, setEdges]);
\end{lstlisting}

\subsection{Theme Implementation}

The system supports both dark and light modes using CSS variables. A theme context provides the current theme state and toggle function. User preference is persisted in localStorage so that the selected theme remains consistent across sessions. CSS variables control node backgrounds, borders, text colors, canvas background, and grid colors, with all components reacting to theme changes. This approach ensures a centralized and maintainable styling system across the application. The dark mode helps reduce eye strain during prolonged use, especially in low-light environments, while the light mode provides a clear interface suitable for documentation and presentations. The theme transition is managed via a React Context provider that updates the global CSS environment, allowing custom components to adapt their internal styling without manual prop drilling. This centralized architecture simplifies the addition of future themes or accessibility adjustments while ensuring theme switching is performed instantly without reloading the page. \textit{Refer Figure~\ref{fig:light-mode}: Same flowchart displayed in light mode}.

\subsection{Export Functionality}

Three export formats are supported: PNG for sharing, SVG for vector graphics, and JSON for data portability. PNG export uses the html-to-image library to capture the canvas as a high-resolution image suitable for documents and presentations. SVG export uses React Flow's built-in rendering to generate scalable graphics that maintain clarity at any resolution. JSON export stores the nodes and edges structure with formatted output, allowing diagrams to be saved, transferred, and re-imported for further editing without losing layout information. \textit{Refer Figure~\ref{fig:export}: Export options dropdown(PNG, SVG, and JSON).}

Providing multiple export formats ensures that the generated diagrams can be used across different environments and workflows. PNG images are suitable for documentation and presentations, SVG files allow further editing in vector graphic tools, and JSON exports enable users to reload and continue editing diagrams later. This flexibility ensures that the system supports both visual sharing and structured data storage.
