\chapter{System Study}
\label{ch:systemstudy}

The system study phase involves understanding existing diagram 
digitization challenges and analyzing requirements for the Automated 
Diagram Digitization system. This chapter covers problem definition, 
feasibility analysis, and requirements gathering. A comprehensive 
investigation was conducted through literature review, analysis of 
existing diagramming tools, and identification of user needs to 
establish a solid foundation for system design and implementation.

%---------------------------------------------------------------
\section{Existing System}
\label{sec:existing-system}
%---------------------------------------------------------------

The current landscape of diagram digitization tools presents several 
significant challenges that limit their effectiveness in real-world 
usage. This section examines the key problems encountered by users 
today and the disadvantages that motivate the development of a more 
intelligent approach.

% C3: Merged Sections 3.1.1 and 3.1.2 into one unified subsection
\subsection{Challenges and Disadvantages of Existing Systems}
\label{sec:existing-challenges}

The existing process of digitizing flowcharts and diagrams faces 
several critical challenges that impact user productivity and diagram 
quality. These challenges arise from the fundamental limitations of 
current tools, which were designed primarily for manual diagram 
creation rather than automated recognition. The following points 
summarize both the operational challenges and the structural 
disadvantages of existing systems, each of which directly motivated 
the design decisions made in the proposed system.

\begin{enumerate}

    \item \textbf{Manual Redrawing Required:} Users must manually 
    recreate diagrams using tools such as Microsoft Visio, Lucidchart, 
    or draw.io -- a time-consuming process that can take 15--30 minutes 
    for moderately complex flowcharts. This not only wastes productive 
    time but also introduces transcription errors in the recreation 
    process.~\cite{flowchart-analysis}

    \item \textbf{Lack of Automated Shape Recognition:} Current 
    systems cannot automatically identify shapes, text, and connections 
    from images of hand-drawn or printed flowcharts. Users must 
    manually identify each shape, enter the label text, and draw 
    connections individually, effectively defeating the purpose of 
    digitization.

    \item \textbf{Text Extraction and Formatting Loss:} Standard OCR 
    tools struggle with handwritten text and cannot preserve line 
    breaks or formatting within diagram shapes. When text spans 
    multiple lines within a shape, the formatting is typically lost, 
    resulting in unreadable merged text that requires manual 
    correction.

    \item \textbf{Connection Detection Gap:} Existing systems fail to 
    recognize the directional relationships between shapes. Even when 
    shapes are detected, the arrows and connections that define the 
    flow logic must be manually drawn -- a particularly significant 
    limitation for complex flowcharts with multiple decision paths and 
    loop-back edges.

    \item \textbf{Limited Input Methods:} Most tools support only 
    manual creation through drag-and-drop interfaces, lacking support 
    for image upload, text-to-flowchart generation from algorithm 
    descriptions, or JSON import for existing flow data.

    \item \textbf{Absence of Intelligent Editing Features:} Automated 
    tools typically lack post-recognition editing capabilities such as 
    auto-layout for clean alignment, undo/redo for error correction, 
    text styling for readability, and edge labeling for decision 
    branches. This limits the usability of any recognized output.

    \item \textbf{Restricted Export Options:} Many tools offer limited 
    export formats, typically only raster image outputs, with no 
    support for SVG vector graphics or JSON data portability. This 
    restricts downstream use of the digitized diagrams across different 
    platforms and workflows.

    \item \textbf{No Theme or Accessibility Support:} Most automated 
    diagramming and recognition tools do not offer theme switching 
    (dark/light mode) or accessibility accommodations, reducing comfort 
    during extended editing sessions.

\end{enumerate}

\section{Proposed System}
\label{sec:proposed-system}
%---------------------------------------------------------------

The Automated Diagram Digitization system is designed to address all 
of the shortcomings identified in the existing system analysis. This 
section provides a high-level overview of the proposed solution and 
outlines the specific advantages it offers over traditional 
approaches.

\subsection{System Overview}
\label{sec:system-overview}

The Automated Diagram Digitization system is a web-based platform 
that combines AI-powered recognition with an interactive diagram 
editor. The system provides a complete workflow from printed diagrams 
to editable digital diagrams:

\begin{enumerate}

    \item \textbf{AI-Powered Recognition:} Uses Google's Gemini AI 
    with custom prompts to analyze uploaded flowchart images, identify 
    shapes (process, decision, start node, input/output), extract 
    embedded text, and recognize connection paths between nodes.

    \item \textbf{Multiple Input Methods:} Supports three input 
    modes -- image upload for existing diagrams, text description for 
    algorithm-to-flowchart conversion, and JSON import for existing 
    flow data.

    \item \textbf{Interactive Diagram Editor:} Provides drag-and-drop 
    from shape palette, undo/redo with snapshot history, auto-layout 
    for clean alignment, node resizing with text adaptation, and edge 
    labeling via double-click.

    \item \textbf{Text Customization:} Enables text formatting 
    including bold, italic, and font size adjustments for node content 
    to enhance readability and presentation quality.

    \item \textbf{Export Functionality:} Supports multiple export 
    formats including PNG for sharing, SVG for vector graphics, and 
    JSON for data portability.

    \item \textbf{Theme Support:} Implements dark/light mode toggle 
    with persistent user preference stored in \texttt{localStorage} 
    for comfortable viewing.

    \item \textbf{Persistent Storage:} Enables saving and retrieving 
    flowcharts using a PostgreSQL database with UUID-based 
    identification and timestamp tracking.

\end{enumerate}

\subsection{Advantages of Proposed System}
\label{sec:advantages}

The following advantages illustrate how the proposed system directly 
resolves the limitations identified in the existing system analysis, 
delivering a more automated, flexible, and feature-rich experience 
for end users.

\begin{enumerate}

    \item Automatic shape identification significantly reduces manual 
    redrawing, saving considerable time and effort per diagram

    \item Text extraction with proper line break preservation maintains 
    readability and original formatting from the source diagram

    \item Automatic connection detection creates edges with correct 
    directional arrows, preserving the logical flow of the original 
    diagram

    \item Multiple input methods provide flexibility for different 
    user workflows and use cases

    \item Comprehensive editing features are comparable to professional 
    diagramming tools, enabling post-recognition refinement

    \item Multiple export options support diverse use cases from 
    documentation to data portability and integration

    \item Undo/redo with snapshot history prevents accidental data loss 
    and enables safe experimentation during editing

    \item Auto-layout ensures a professional appearance with properly 
    aligned diagrams after recognition or manual editing

    \item Theme switching (dark/light mode) reduces eye strain during 
    extended editing sessions

    \item Persistent storage enables workflow continuity across 
    separate user sessions

\end{enumerate}

\section{Feasibility Study}
\label{sec:feasibility}
%---------------------------------------------------------------

A thorough feasibility study was conducted to evaluate whether the 
proposed system can be successfully developed and deployed within the 
available resources. The study examines technical, operational, and 
economic dimensions to confirm that the project is both achievable 
and sustainable.

\subsection{Technical Feasibility}
\label{sec:tech-feasibility}

The project utilizes proven, mature technologies that have 
demonstrated reliability across numerous deployments. Each component 
was selected based on its stability, community support, and 
suitability for the specific requirements of the platform. The 
combination of FastAPI, React, PostgreSQL, and Google Gemini AI forms 
a well-integrated technology stack that minimizes implementation risk 
while maximizing developer productivity.

\begin{enumerate}

    \item \textbf{Backend:} FastAPI provides high-performance 
    asynchronous API endpoints with automatic OpenAPI documentation, 
    making development and testing efficient.~\cite{fastapi}

    \item \textbf{AI Integration:} Google Gemini API offers 
    state-of-the-art multimodal capabilities for image analysis and 
    text generation, with extensive documentation and community 
    support.~\cite{gemini}

    \item \textbf{Frontend:} React~18 with Vite provides optimal 
    performance with fast refresh and efficient rendering, while React 
    Flow offers specialized diagramming capabilities.~\cite{react}

    \item \textbf{Database:} PostgreSQL with JSONB support provides 
    robust data persistence with ACID compliance and a flexible schema 
    for storing complex flowchart data.~\cite{postgresql}

    \item \textbf{Deployment:} The modular architecture supports cloud 
    deployment with horizontal scaling capabilities as user numbers 
    grow.

\end{enumerate}

Each technology in the stack has extensive community support and a 
demonstrated track record in application development, collectively 
reducing implementation risk.

\subsection{Operational Feasibility}
\label{sec:op-feasibility}

Beyond technical considerations, the system must be operationally 
feasible -- that is, practically usable by the intended audience with 
minimal training or specialized knowledge. The following points 
outline how the platform design ensures smooth day-to-day operation.

\begin{enumerate}

    \item \textbf{User-Friendly Interface:} The input page features a 
    tabbed interface with clear options for image upload, text prompt, 
    and JSON import, requiring minimal user training.

    \item \textbf{Intuitive Editor:} Drag-and-drop from shape palette, 
    familiar undo/redo shortcuts (\texttt{Ctrl+Z}, \texttt{Ctrl+Y}), 
    and double-click edge labeling follow common software conventions.

    \item \textbf{Clear Feedback:} Visual indicators show upload 
    progress, processing status, and error messages with actionable 
    guidance.

    \item \textbf{Minimal Maintenance:} The system is designed for 
    automatic error recovery with fallback mechanisms for malformed AI 
    responses.

    \item \textbf{Responsive Design:} The interface adapts to 
    different screen sizes, ensuring usability on desktop, tablet, 
    and mobile devices.

\end{enumerate}

\subsection{Economic Feasibility}
\label{sec:economic-feasibility}

The economic feasibility of the proposed system was assessed by 
examining the costs associated with development, deployment, and 
ongoing maintenance. The analysis confirms that the project can be 
delivered within a reasonable budget and provides measurable returns 
in terms of time savings and productivity gains.

\begin{enumerate}

    \item \textbf{Open-Source Technologies:} FastAPI, React, 
    PostgreSQL, and related libraries are open-source with no 
    licensing costs, minimizing the initial development investment.

    \item \textbf{Cost-Effective AI:} Google Gemini API offers 
    competitive pricing with free-tier options suitable for 
    development and testing phases.

    \item \textbf{Reduced Labour Costs:} Automation of diagram 
    digitization significantly reduces hours of manual redrawing, 
    providing measurable time savings for users.

    \item \textbf{Scalable Architecture:} The system can be deployed 
    on cloud infrastructure with pay-as-you-grow pricing, avoiding 
    large upfront infrastructure costs.

    \item \textbf{Return on Investment:} Educational institutions and 
    businesses can achieve positive ROI through improved productivity 
    and reduced manual digitization effort.

\end{enumerate}

\section{Requirements Analysis}
\label{sec:requirements}
%---------------------------------------------------------------

Requirements analysis is a critical phase that bridges the gap 
between problem identification and system design. It involves 
identifying, documenting, and organizing the specific capabilities 
and constraints that the system must fulfill. The requirements for 
the Automated Diagram Digitization system were gathered through 
analysis of existing tools, evaluation of user needs, and review of 
the challenges identified earlier in this chapter.

\subsection{Functional Requirements}
\label{sec:functional-req}

Based on user needs and system objectives, the following functional 
requirements have been identified. These requirements define the core 
capabilities the system must provide in order to successfully 
digitize diagrams and allow users to interact with them efficiently.

\subsubsection{Input Module Requirements}

The input module is responsible for collecting diagram data from the 
user in various forms and preparing it for processing. The following 
requirements define the accepted input formats, validation rules, and 
interface behaviors that must be supported.

\begin{enumerate}

    \item \textbf{Image Upload:} Support PNG, JPG, and WebP image 
    formats with drag-and-drop interface, file size validation, and 
    preview functionality.

    \item \textbf{Text Prompt:} Provide a multi-line textarea for 
    algorithm descriptions with a character count indicator and 
    example placeholder text.

    \item \textbf{JSON Import:} Accept JSON file upload with 
    validation and descriptive error messaging for malformed data.

\end{enumerate}

\subsubsection{AI Processing Requirements}

The AI processing module handles communication with Google Gemini AI 
to perform shape recognition, text extraction, and connection 
detection. The following requirements ensure that AI interactions 
are accurate, robust, and capable of producing well-structured 
flowchart data.

\begin{enumerate}

    \item \textbf{Shape Recognition:} Identify four basic shape 
    types -- process (rectangle), decision (diamond), start node 
    (oval shape), and input/output (parallelogram) -- from uploaded 
    images.

    \item \textbf{Text Extraction:} Extract text from shapes with 
    proper line break preservation using \texttt{\textbackslash n} 
    characters for multi-line labels.

    \item \textbf{Connection Detection:} Detect edges between nodes 
    with directional arrows and assign appropriate connection handles 
    (top, bottom, left, right).

    \item \textbf{Coordinate Estimation:} Calculate normalized 
    coordinates on a 0--1000 scale for consistent node positioning 
    across images of different sizes.

    \item \textbf{Error Handling:} Parse AI responses reliably with 
    a regex-based fallback mechanism for extracting JSON from mixed 
    text responses.

\end{enumerate}

\subsubsection{Editor Module Requirements}

The editor module provides an interactive canvas for users to view 
and modify the generated flowchart. It must support a rich set of 
interactions -- including node manipulation, edge editing, styling, 
and layout management -- to deliver a professional diagram editing 
experience.

\begin{enumerate}

    \item \textbf{Drag-and-Drop Functionality:} Allow users to drag 
    shapes from the sidebar palette and drop them onto the canvas at 
    desired positions.

    \item \textbf{Node Resizing:} Support node resizing with minimum 
    dimension constraints and automatic text adaptation to fit resized 
    nodes.

    \item \textbf{Undo/Redo:} Implement snapshot-based history with 
    keyboard shortcuts (\texttt{Ctrl+Z} for undo, \texttt{Ctrl+Y} or 
    \texttt{Ctrl+Shift+Z} for redo).

    \item \textbf{Auto-Layout:} Provide one-click auto-layout using a 
    column-based clustering algorithm for clean diagram alignment.

    \item \textbf{Edge Labeling:} Enable double-click on edges to add 
    and edit labels with proper background styling for readability.

    \item \textbf{Deletion:} Support deletion of selected nodes and 
    edges using the \texttt{Delete} or \texttt{Backspace} keys with 
    automatic snapshot capture.

    \item \textbf{Color Customization:} Allow users to change node 
    and edge colors through a color picker interface.

    \item \textbf{Text Styling:} Provide controls for bold, italic, 
    and font size adjustments for the content of selected nodes.

\end{enumerate}

\subsubsection{Export Module Requirements}

The export module enables users to save or share their completed 
diagrams in formats suitable for various professional and academic 
contexts.

\begin{enumerate}

    \item \textbf{PNG Export:} Capture the entire canvas as a PNG 
    image using the \texttt{html-to-image} library.

    \item \textbf{SVG Export:} Generate SVG vector graphics using 
    React Flow's built-in SVG renderer.

    \item \textbf{JSON Export:} Serialize the current nodes and edges 
    to a JSON file with proper indentation for readability.

\end{enumerate}

\subsubsection{Storage Module Requirements}

The storage module manages the persistence of flowchart data across 
user sessions, ensuring that diagrams can be saved, retrieved, and 
updated reliably.

\begin{enumerate}

    \item \textbf{Save Flowchart:} Store flowchart data in the 
    PostgreSQL database with a UUID primary key and timestamp 
    tracking.

    \item \textbf{Retrieve Flowchart:} Fetch a flowchart by UUID and 
    reconstruct the editor state from stored data.

    \item \textbf{Update Flowchart:} Modify an existing flowchart's 
    title or flow data, with automatic refresh of the 
    \texttt{updated\_at} timestamp.

\end{enumerate}

\subsection{Non-Functional Requirements}
\label{sec:nonfunctional-req}

Non-functional requirements define the quality attributes of the 
system rather than its specific behaviors. They describe how well 
the system performs its functions, covering aspects such as speed, 
usability, reliability, and security, and are essential for ensuring 
a positive user experience and long-term system sustainability.

\subsubsection{Performance Requirements}

\begin{enumerate}

    \item \textbf{Image Analysis:} Complete image analysis within 
    3--5 seconds for standard flowchart images up to 
    1024$\times$1024 pixels.

    \item \textbf{Text-to-Flowchart:} Generate a flowchart from a 
    text prompt within 2--3 seconds.

    \item \textbf{Editor Operations:} Execute node drag-and-drop, 
    undo/redo, and auto-layout within 100\,ms for a smooth user 
    experience.

    \item \textbf{API Response:} Maintain API response times under 
    200\,ms for standard database operations.

    \item \textbf{Page Load:} Achieve initial page load in under 
    2 seconds with optimized asset delivery.

    \item \textbf{Concurrent Users:} Support at least 50 concurrent 
    users without performance degradation.

\end{enumerate}

\subsubsection{Usability Requirements}

\begin{enumerate}

    \item \textbf{Intuitive Interface:} Design an interface requiring 
    minimal training for users with basic computer skills.

    \item \textbf{Clear Error Messages:} Provide actionable error 
    messages with guidance for resolution.

    \item \textbf{Responsive Design:} Ensure comfortable usage on 
    desktop monitors, tablets, and mobile phones.

    \item \textbf{Theme Consistency:} Maintain a consistent 
    appearance across dark and light themes with smooth transitions.

    \item \textbf{Helpful Tooltips:} Include tooltips for all toolbar 
    buttons explaining their functionality.

    \item \textbf{Keyboard Shortcuts:} Support common keyboard 
    shortcuts for undo/redo and delete operations.

\end{enumerate}

\subsubsection{Reliability Requirements}

\begin{enumerate}

    \item \textbf{Uptime:} Maintain 99\% availability during normal 
    business hours.

    \item \textbf{Error Recovery:} Implement graceful error handling 
    with fallback mechanisms for AI processing failures.

    \item \textbf{Data Persistence:} Ensure data integrity with 
    transaction support for all database operations.

    \item \textbf{Input Validation:} Validate all user inputs on both 
    the client and server sides to prevent invalid data from reaching 
    the AI or database layers.

    \item \textbf{Backup:} Support database backup and restore 
    procedures for disaster recovery.

\end{enumerate}

\subsubsection{Security Requirements}

\begin{enumerate}

    \item \textbf{API Key Protection:} Store the Gemini API key 
    exclusively in server-side environment variables; never expose it 
    in client-side code.

    \item \textbf{Input Sanitization:} Sanitize all user inputs to 
    prevent injection attacks.

    \item \textbf{CORS Configuration:} Restrict API access to allowed 
    origins (\texttt{localhost} for development, the verified 
    production domain for deployment).

    \item \textbf{SQL Injection Prevention:} Use parameterized queries 
    for all database operations via SQLAlchemy.

    \item \textbf{Environment Separation:} Maintain separate 
    configuration files for development, testing, and production 
    environments.

\end{enumerate}

\section{System Modules}
\label{sec:modules}
%---------------------------------------------------------------

The Automated Diagram Digitization platform is organized into five 
functional modules, each responsible for a specific part of the 
overall workflow. This modular design simplifies development and 
maintenance while allowing individual components to be improved 
independently.

\subsection{Input Module}

The input module serves as the primary entry point of the system, 
accepting user-provided data in multiple forms and preparing it for 
further processing by the AI engine and the diagram editor. It 
handles three input methods: image upload with drag-and-drop 
interface, text prompt with multi-line textarea, and JSON import 
with file upload. It validates input formats, provides visual 
feedback, and prepares data for AI processing.

\subsection{AI Processing Module}

The AI processing module coordinates communication with the external 
Gemini AI service, managing both outgoing requests and the parsing 
of structured responses returned by the model. It interfaces with 
Google Gemini AI using custom system prompts for image analysis and 
text-to-flowchart conversion, parses JSON responses, handles errors 
with a regex-based fallback, and ensures proper ID generation for 
edges.

\subsection{Editor Module}

The editor module forms the interactive core of the platform, 
enabling users to view, manipulate, and refine the flowcharts 
generated by the AI processing module. It renders nodes and edges 
using React Flow with custom node components for each shape type, 
and implements drag-and-drop from sidebar, undo/redo with snapshot 
history, auto-layout, edge labeling via double-click, and 
comprehensive styling options.

\subsection{Export Module}

The export module provides users with the ability to save and 
distribute their diagrams in formats suited to a range of 
professional and educational use cases. It provides PNG export using 
the \texttt{html-to-image} library, SVG export using React Flow's 
SVG renderer, and JSON export of the current flow state, with file 
downloads named according to the flowchart title.

\subsection{Storage Module}

The storage module ensures long-term persistence of diagram data, 
allowing users to save their work and retrieve it in future sessions 
without loss of layout or content information. It manages PostgreSQL 
database operations including save, retrieve, and update of 
flowcharts, implementing UUID-based identification, timestamp 
tracking, and JSONB storage for complete flow data.

\section{Conclusion}
\label{sec:ch3-conclusion}
%---------------------------------------------------------------

The system study has established clear and comprehensive requirements 
for the Automated Diagram Digitization project, addressing critical 
gaps in automated shape recognition, text extraction, connection 
detection, and interactive editing. The analysis of existing systems 
revealed significant inefficiencies in manual redrawing workflows, 
the absence of automation, and limited editing capabilities in 
current tools. These limitations highlight the need for a system 
that can intelligently interpret diagram structures while also 
providing flexible post-recognition editing capabilities.

The proposed system directly addresses these challenges through 
AI-powered recognition, multiple input methods, comprehensive editing 
features, and flexible export options. Feasibility analysis confirms 
that the solution is technically achievable using proven technologies 
such as FastAPI, React, PostgreSQL, and Google Gemini AI, which 
collectively provide a stable and scalable foundation.

From an operational perspective, the system offers a user-friendly 
interface that simplifies the overall workflow from diagram input to 
final export. Economically, the use of open-source technologies and 
cloud-based infrastructure reduces development and operational costs 
while providing long-term sustainability. Chapter~4 presents the 
detailed system design, translating the requirements established here 
into architectural, data flow, and module-level design 
specifications.
